# ADR-024: `--debug` prints resolved profile + auth source, never a raw credential

## Decision
`--debug` prints two lines to stderr before a command runs: the resolved profile name + URL, and which credential path is active (`OAuth bearer token` or `api_key:api_secret from profile file`). It never prints `api_key`, `api_secret`, `access_token`, or `refresh_token` values. Implemented in the main app/verb router only — not the `mcp`, `auth`, or `profile` sub-command paths.

## Context
Chctl-inspired (`ROADMAP.md` Functional bucket) — chctl's `cloud --debug` prints resolved credential source and API URL to stderr before running, to help debug multi-profile/credential-precedence confusion. frappe-ctl's `config.ts` already has an equivalent precedence (`--site` override → `cfg.default`, then OAuth bearer takes priority over `api_key:api_secret` per ADR-009) that was previously invisible to the operator.

ADR-020 (credential leak boundary) explicitly pre-constrained this feature before it existed: *"Any future `--debug`/`--verbose` flag MUST print only the credential source, never the value."* `debugInfo()` in `cli.ts` is a pure, exported function (same pattern as `isVerbAllowed`/`exitCodeFor`) so this constraint is regression-tested directly — `cli.test.ts` asserts the output never contains the configured `api_key`/`api_secret`/`access_token`/`refresh_token` values, mirroring the guard already in place for `client.ts` error messages (ADR-020).

Scoped to the main verb router only. `mcp`, `auth`, and `profile` sub-commands don't build a `FrappeClient` the same way (or don't need one at all), so wiring `--debug` into each would be three more call sites for comparatively low value — the main router is where the profile/auth-precedence confusion chctl's feature targets actually shows up.

## Consequences
- ✅ Surfaces existing but previously invisible precedence logic (`--site` override, OAuth-over-api_key priority) without adding a new precedence system
- ✅ Regression-tested against the exact leak scenario ADR-020 anticipated
- ✅ `debugInfo()` is pure and testable without capturing `console.error` or spawning a subprocess
- ⚠️ Does not report Keychain-vs-file-store distinction for the OAuth token path — `token-store.ts::loadToken()` doesn't currently surface which backend actually served the token, and adding that is out of scope here
- ⚠️ Not wired into `mcp`/`auth`/`profile` sub-commands — if profile-resolution confusion turns out to matter there too, extend `debugInfo()`'s call sites rather than duplicating the logic
